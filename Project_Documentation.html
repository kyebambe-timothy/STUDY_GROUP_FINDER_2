1. System Architecture Diagram

The "Student Study Group Finder" follows a robust three-tier application architecture. This structure segregates the responsibilities of the system to ensure modularity, scalability, and ease of maintenance in line with modern web development standards.

Client Tier (Frontend): Built using React.js and Vite. It utilizes React Router for seamless single-page application navigation and Axios for executing secure HTTP requests to the backend API.
Application Tier (Backend): Developed with Node.js and Express.js, providing robust RESTful APIs. It manages secured routing (authentication, user management, and study groups) and communicates with the database using the Sequelize Object-Relational Mapper (ORM).
Data Tier (Database): A MySQL relational database responsible for persistently storing user credentials, group memberships, and session data, bounded by strict constraints and relationships.

2. Entity-Relationship (ER) Diagram

The database structure relies on normalized relational data modeling to prevent data duplication and maintain referential integrity. User objects, Study Group entities, Group Memberships, and Study Sessions interact dynamically through properly enforced Foreign Keys.

The ER diagram below conceptualizes the structural bindings between the core modules. Users can belong to multiple groups (Many-to-Many resolved by the GroupMember junction table), and study groups can hold multiple scheduled sessions or posts.

3. User Manual

The Student Study Group Finder is designed to offer an intuitive user experience for seamless academic collaboration across the university campus. This section provides operational guidelines for core functionality.

3.1. Account Registration & Authentication

Upon loading the platform, unregistered students must select the Register option. They are required to input their Full Name, valid Email address, a secure Password, their Program of Study, and Year of Study. Data validation guarantees inputs meet requirements. Existing users can access their secured dashboard via the Login portal leveraging JWT-based authentication.

3.2. Discovering and Joining Study Groups

After a successful login, the user's dashboard displays a repository of active study groups.

  • Search/Filter: Students can query groups efficiently based on Course Name, Group Title, or Faculty.
  • Group Details: Clicking on a group reveals its comprehensive details, such as group description, course focus, meeting medium (Virtual or Physical location), and capacity constraints.
  • Enrollment: If the parameters are favorable, users select the "Join Group" option, instantly converting them into active members.

3.3. Creation & Administration of Study Groups

Students possess the autonomy to initiate study groups tailored to specific academic needs.

  • Initialization: From the dashboard, selecting "Create Group" prompts a form requiring the Group Name, pertinent Course Code, Description, and the default Meeting Location.
  • Leadership Capabilities: The user initiating the group assumes the system role of a Group Leader. Leaders have escalated privileges to modify group metadata, evaluate member activity, and arbitrarily expel inactive peers.

3.4. Orchestrating Study Sessions

Organized scheduling is instrumental. Group leaders can invoke the "Create Session" capability directly from the group interface. This schedules upcoming agendas and broadcasts the Date, specific Time, localized or Web Meeting link, and a descriptive brief to all group members. Members monitor upcoming iterations via their personal Dashboard.

3.5. Internal Communication

In-group dialogues cultivate learning. Each group inherently manages a designated communication board where members can dispatch short posts to inquire about challenging topics, circulate notes, or align on upcoming scheduled sessions proactively.

4. API Documentation

The system exposes a secure REST API architecture for client-server communication. Secured endpoints necessitate a valid JSON Web Token generated upon login, supplied through the HTTP Authorization header context (`Bearer <TOKEN>`).

4.1 User Authentication Module

Method / Endpoint Description & Specification
POST
/api/auth/register		 Description: Registers a user identity within the system. Payload constraints are verified.
Payload Schema: { username: String, email: String, password: String, role: String, programOfStudy: String }
POST
/api/auth/login		 Description: Authenticates the user credentials and returns an encrypted JWT payload.
Payload Schema: { email: String, password: String }

4.2 Study Group Administration

Method / Endpoint Description & Specification
GET
/api/groups		 Description: Fetches all formally registered study groups. Supports query string filters (e.g. ?faculty=Science).
POST
/api/groups		 Description: Provisions a new study group object. Assigns the requester as the Group Leader.
